# 📄 产品需求文档 (PRD)：简易在线商城系统 (V1.0)

**撰写人：** 产品经理 / 全栈开发者

**目标技术栈：** 后端 Spring Boot (Java) + 前端 Vue 3 + MySQL + Redis

## 1. 产品概述

### 1.1 项目背景与目标

本项目旨在从零搭建一个具备完整交易闭环的在线商城。V1.0 版本（MVP）将剔除复杂的分销、多店铺入驻等功能，聚焦于**自营电商**最核心的业务：商品展示、购物车、下单支付、后台发货与库存管理。

目标是快速跑通业务主干，验证前后端交互（尤其是在高并发下的库存防超卖机制）与支付链路的可行性。

### 1.2 目标用户群体

- **C端（消费者）：** 需要在移动端/PC端浏览商品、比价、并完成购买的用户。
- **B端（商家/运营人员）：** 负责上架商品、处理订单、管理库存及查看财务数据的内部员工。

------

## 2. 整体业务流程

在进入具体功能前，团队所有成员必须清晰理解电商的核心交易链路。

**核心交易主流程：**

用户登录 -> 浏览商品页 -> 选择具体的商品规格 (SKU) -> 加入购物车 -> 提交订单 (系统预扣库存) -> 调起支付网关 -> 支付成功 (系统确认库存，生成正式订单) -> 商家后台发货 -> 用户确认收货。

------

## 3. 核心功能需求 (C端 - 消费者商城)

### 3.1 用户认证模块

#### 环节 A：用户注册 (Register)

1. **前端 (Vue)：** 用户输入 `username` 和 `password`，点击注册。前端可以做个简单校验（比如密码长度不能小于 6 位）。
2. **后端 (Spring Boot)：**
   - 接收到请求后，先去数据库查一下 `username` 是否已经存在，存在则抛出异常“用户名已被注册”。
   - **核心安全操作：** 使用 Spring Security 提供的 `BCryptPasswordEncoder` 将前端传来的明文密码进行加密。
   - 将 `username` 和**加密后的密文密码**存入数据库。

#### 环节 B：用户登录 (Login)

1. **前端 (Vue)：** 用户输入 `username` 和 `password`，点击登录。
2. **后端 (Spring Boot)：**
   - 根据 `username` 去数据库查出这个用户的信息（此时拿到的是加密后的密码）。
   - 如果用户不存在，返回“用户名或密码错误”。
   - **核心比对操作：** 使用 `BCryptPasswordEncoder.matches(明文密码, 数据库里的密文密码)` 方法进行比对。
   - **颁发令牌：** 如果比对成功，后端根据该用户的 ID 和 Username 生成一个 **JWT (JSON Web Token)**，并返回给前端。
3. **前端 (Vue) 后续操作：** 前端拿到 JWT 后，将其存入 `localStorage` 或 `Pinia` 状态管理中。之后用户去浏览商品详情、加购物车、下单时，前端都要在 HTTP 请求的 `Header` 里带上这个 Token，后端通过解析 Token 就能知道是哪个用户在操作。

### 3.2 商品浏览与详情模块

- **功能描述：** 展示商城内所有可售商品，并提供规格选择。
- **业务逻辑（极其关键）：**
  - **SPU 展示：** 商品列表页展示商品的通用信息（如主图、标题、最低指导价）。
  - **SKU 联动：** 用户进入详情页后，必须选择完整的规格（如：颜色+尺码）。前端根据用户选择，实时向后端查询该特定 SKU 的**准确库存**和**实际价格**。
- **异常处理：** 如果某个 SKU 库存为 0，前端该规格按钮必须置灰，显示“缺货”，并禁用“加入购物车”按钮。

### 3.3 交易链路与订单模块

- **功能描述：** 购物车管理、结算页信息确认、订单生成与状态流转。
- **业务逻辑：**
  - **防超卖机制（核心技术难点）：** 用户点击“提交订单”时，后端必须使用 Redis 或数据库乐观锁进行库存预扣减。
  - **订单状态机：** 订单生命周期严格按照下表流转。

| **当前状态** | **触发动作**            | **流转后状态** | **备注逻辑**                           |
| ------------ | ----------------------- | -------------- | -------------------------------------- |
| **待支付**   | 用户提交订单            | 待支付         | 锁定库存。**系统开启 15 分钟倒计时**。 |
| 待支付       | 用户完成付款            | **待发货**     | 记录支付流水号，真正扣减有效库存。     |
| 待支付       | 15分钟未付款 / 手动取消 | **已取消**     | 订单关闭，**释放预扣的库存**。         |
| 待发货       | B端商家填写物流发货     | **待收货**     | 更新物流单号，用户可查看物流状态。     |
| 待收货       | 用户点击确认收货        | **已完成**     | 交易闭环结束。                         |

## 4. 核心功能需求 (B端 - 商家管理后台)

### 4.1 商品与库存管理

- **功能描述：** 运营人员进行商品信息的录入和上下架。
- **业务逻辑：**
  - **基础信息：** 填写 SPU 名称、类目、上传富文本详情页。
  - **规格矩阵录入：** 运营添加属性（如颜色：红/白；容量：64G/128G）后，前端使用 Vue 动态生成表格，要求运营必须为每一个交叉生成的 SKU 设置**独立的价格、库存量和 SKU 编码**。
- **异常处理：** 已有未完成订单的商品，严禁直接删除，只能操作“下架”。

### 4.2 订单与履约管理

- **功能描述：** 处理每日订单，进行发货操作。
- **业务逻辑：**
  - 支持按订单号、手机号、订单状态进行复合查询。
  - **发货操作：** 针对“待发货”订单，运营需输入快递公司与快递单号，点击发货后，系统向 C 端用户推送发货通知。

### 4.3 数据看板 (Dashboard)

- **功能描述：** 提供商城的宏观运营数据呈现。
- **指标需求：** * 核心指标：今日 GMV（交易总额）、今日实付订单数、今日新增注册用户数。
  - 列表展示：库存预警列表（展示库存少于 10 件的 SKU 供补货参考）。

## 5. 非功能性与技术规范需求

作为技术驱动的 PRD，必须对研发提出系统级的硬性要求，以匹配 Spring Boot + Vue 的企业级定位：

1. **事务一致性保障：** * **要求：** 在“提交订单”接口中，涉及“扣库存、扣优惠、写订单表、写明细表”等多个数据库写操作，必须使用 Spring 的 `@Transactional` 保证 ACID 特性。一旦发生异常，全部回滚，**绝不允许出现“钱扣了但没订单”或“订单生成了但库存没扣”的数据污染**。
2. **接口安全性与防刷：** * **要求：** 核心交易接口（获取验证码、提交订单、支付回调）必须接入 Spring Boot 层的限流组件（如整合 Redis 的 Token Bucket 算法），防止恶意脚本瞬间并发打挂数据库。
3. **前后端数据类型强制约束：**
   - **要求：** 前端（哪怕不用 TypeScript）在向后台传递金额相关数据时，严禁使用 JavaScript 原生的浮点数直接计算（避免 `0.1 + 0.2 = 0.30000000000000004` 问题）。**金额在前后端传输及数据库存储（MySQL）时，建议统一使用 `整数`（单位：分）或 `DECIMAL` 类型。**
